# CTO Audit: bndy Brain Alignment Review

**Date:** 03/05/2026  
**Reviewer:** ChatGPT acting as BNDY CTO reviewer  
**Scope:** `florence` / bndy-brain, `bndy-signals`, `bndy-frontstage`  
**Status:** Action required before continuing source automation

---

## Executive Summary

bndy is trying to become an AI-native grassroots live music knowledge system, not a traditional gig submission app.

The intended model is:

```text
Evidence in
→ claims generated internally
→ candidate entities and graph proposals
→ human/chat ratification
→ durable graph and memory
→ better future reasoning
```

The Brain documentation broadly captures this intent. However, the current implementation and active backlog are drifting toward a more mechanical claims-processing system with a chat-shaped input surface.

The risk is that bndy becomes a claims review/admin workflow instead of an agentic knowledge system.

---

## What bndy Brain Must Be

`bndy-brain` should be the canonical operating memory for all human and AI agents working on the product.

It must explain:

- what bndy is building
- why it is being built this way
- how the Brain works
- how evidence becomes knowledge
- what agents must optimise for
- what not to build
- where the architecture is intentionally different from legacy CRUD/admin systems

The Brain is not simply a documentation repo. It is the product's durable reasoning layer.

If an agent is unsure, it should be able to read this repo and understand the product direction without relying on prior chat context.

---

## What Is Correct

The Brain docs already contain the right core philosophy.

The correct conceptual model is:

```text
raw signal → model interprets context → proposes world-state change → evidence is retained → human/agent accepts, rejects or challenges → memory updates
```

The docs correctly state that:

- signals are long-lived evidence
- interpretations are versioned understanding
- claims are proposed world-state changes
- graph updates should be proposed, not blindly written
- humans correct reasoning through challenge loops
- evidence accumulates over time
- knowledge should remain explainable

This is directionally right and should remain the architectural north star.

---

## Current Drift

The active implementation is not yet enforcing the Brain model.

### Current implementation pattern

```text
Signal
→ interpretation
→ claims
→ review UI
→ accepted claim
→ fuzzy entity match
→ create/link canonical entity
```

This is mechanically useful, but it is not yet the intended Brain architecture.

### Intended Brain pattern

```text
Signal or chat/poster submission
→ claims generated internally
→ Brain creates candidate entities/proposals
→ chat asks targeted ratification questions
→ human confirms/corrects ambiguity
→ evidence pack / graph updated
→ canonical knowledge evolves
```

The difference matters.

The current implementation treats claims as the main user-facing review unit. The intended model treats claims as internal evidence and reasoning artefacts. The user should mostly interact with natural confirmation questions, not raw claim rows.

## Chat Review

The `/chat` page is now runtime-compatible with the current signals API.

It correctly:

- submits as `text_paste`
- reads `llmInterpretation.reasoning`
- uses environment-based API configuration
- uses a textarea for multi-line input

However, it is not yet a true Brain chatbot.

It currently behaves as:

```text
chat text → text_paste signal → poll result → display reasoning summary
```

It does not yet:

- preserve meaningful conversation context
- ask targeted follow-up questions
- present candidate entity matches
- resolve ambiguity such as "Which Rigger?"
- apply user corrections back into claims/entities
- call claim review APIs conversationally
- update the graph through ratification
- keep claims hidden while using them internally

Therefore `/chat` should be documented as **Signal Chat v0**, not as completed conversational intelligence.

## BUILD-004 Review

BUILD-004 introduced useful mechanical plumbing:

- canonical artist / venue / event schemas
- claim-to-entity linking
- fuzzy name matching
- entity resolver integration with accepted claims

But it is not sufficient for Brain-grade entity resolution.

Known issues:

| Issue | Risk |
|------|------|
| Event drafts cast through `unknown` | Type safety bypassed; schema-invalid event drafts possible |
| Edited claims ignored during resolution | User corrections may not affect entity creation/linking |
| Fuzzy name-only matching | High risk of incorrect venue/artist merges |
| No location-aware venue resolution | Common venue names will collide across towns |
| No external reference model | Google/Facebook/Instagram/ticketing identifiers cannot be retained as evidence |
| No candidate entities | Ambiguity is hidden rather than surfaced |
| No confidence/corroboration reasoning | System cannot explain why it linked entities |
| No chat-driven ratification | Claims UI remains required |

BUILD-004 should be reclassified as **mechanical draft entity linking**, not full entity resolution.

---

## Source Profiles Should Not Be Next

The current backlog lists Source Profiles as the next BUILD item.

That is premature.

Source Profiles will increase the volume of signals. If implemented before candidate resolution and ratification are safe, they will scale noise, duplicate entities, and bad merges.

Do not automate ingestion before the Brain can:

- identify candidate entities
- explain match reasoning
- surface ambiguity
- ask humans for ratification
- preserve external references as evidence
- update the graph safely

Automation should come after safe reasoning, not before it.

---

## Required Reframe

The next build should not be Source Profiles.

The next build should be:

```text
BUILD-005: Conversational Entity Ratification
```

Purpose:

> Convert generated claims into candidate graph updates and use chat to ratify ambiguity before canonical entities are created or linked.

Target flow:

```text
User submits poster/text/chat message
→ Signal created
→ Interpretation generates claims internally
→ Brain compares claims to memory
→ Candidate entities and graph proposals created
→ Chat asks confirmation/correction questions
→ Human answers naturally
→ Evidence pack and graph update
→ Claims remain hidden from normal users
```

Example:

```text
bndy: I think this is Stingray playing at The Rigger on 15 May at 8pm.
      I found more than one possible venue called The Rigger.
      Is this The Rigger in Newcastle-under-Lyme?

User: Yes, Newcastle.

bndy: Great. I have linked this event to The Rigger, Newcastle-under-Lyme and added the poster as supporting evidence.
```

---

## Required Architecture Decisions

Create an ADR before further build work continues.

Suggested ADR:

```text
ADR-004: Conversational Entity Ratification
```

It should state:

- claims are internal reasoning artefacts, not the primary user UI
- BNDY canonical IDs are the primary identity
- external IDs such as Google Place ID are evidence references, not canonical identity
- entity resolution produces candidates/proposals before truth
- chat is the human ratification layer
- evidence packs are the bridge between raw claims and canonical entities
- canonical entities should not be created or linked blindly when ambiguity exists
- source automation must wait until ratification is safe

## Required BUILD-005 Scope

BUILD-005 should create the missing Brain layer.

### Core concepts

```typescript
interface EntityCandidate {
  candidateId: string;
  entityType: 'artist' | 'venue' | 'event';
  proposedName: string;
  matchedEntityId?: string;
  matchReasoning: string;
  ambiguityReason?: string;
  evidenceClaimIds: string[];
  externalRefs?: ExternalReference[];
  status: 'proposed' | 'confirmed' | 'rejected' | 'needs_clarification';
}

interface ExternalReference {
  source: 'google_places' | 'facebook' | 'instagram' | 'ticketing' | 'website' | 'user_submission';
  externalId?: string;
  url?: string;
  confidence: 'weak' | 'moderate' | 'strong';
  reasoning: string;
}
```

### Behaviour

- Generate candidates from claims.
- Do not auto-link ambiguous names.
- Ask chat questions for ambiguity.
- Capture user answers as evidence/challenges.
- Update evidence packs before canonical entities.
- Keep claims out of the normal user-facing UI.

## Product Principle

Do not build a better form.

Do not build a claims admin tool with chat bubbles.

Build a system that can say:

```text
I believe this gig exists.
Here is why.
Here is what I am unsure about.
Please confirm the ambiguous bit.
I will remember your correction for next time.
```

That is the product.
